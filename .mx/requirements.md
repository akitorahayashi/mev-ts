# Refactor Instructions: mev-ts Technical Debt Reduction

## Goal

The mev-ts codebase carries a bounded set of technical debt: a handful of latent correctness defects, copy-pasted structures that force multi-site edits, three misplaced unit tests that break the hermetic-unit-suite guarantee, and a few design-level fragilities. The desired outcome is a codebase where each of the confirmed defects is fixed, each duplicated structure has a single owner, the test suite enforces its own layer rules, and the decided design changes (Group D; user decisions recorded 2026-07-20) are implemented. The single deliberately rejected redesign (D2, dispatch descriptor table) stays as a recorded decision, not code.

## Current State

### What the project does

`mev` is Local IaC for macOS, built with Bun and TypeScript, compiled to a single binary. The repository config is the source of truth for personal machine setup. Provisioning runs three sequential phases per request: deploy embedded role assets to `~/.mev/roles/{role}/`, install missing Homebrew packages via Brewfile batch, then apply declared activations (symlinks, macOS defaults, duti associations, pipx tools, editor extensions, Coder/Zed config rendering, host-command pipelines, release-binary downloads, remote installers).

### Key user workflows

- `mev make <target...>` (alias `mk`) provisions explicit targets.
- `mev create` (alias `cr`) provisions every non-optional registered target.
- `mev sync` (alias `s`) scans for semantic drift (target-signature mismatch or deployed-role drift) and re-provisions only stale targets. Applied signatures live under `~/.mev/applied/{target}`.
- `mev config agents|skills|zed` toggles selection manifests interactively.
- `mev user show|set` and `mev switch <scope>` manage Git identities stored in `~/.mev/identity.json`.
- `mev list` renders the target table; hidden `mev internal ...` commands back the `md2pdf`/`pdf2md` shell aliases and gh/git maintenance operations.

### Entry points and module responsibilities

- `src/main.ts` builds the clipanion CLI from `src/cli/commands/registry.ts` (single registration source, 19 command classes) and rewrites `namespace --help` arguments.
- `src/provisioning/run.ts` — `runMake()` — is the 3-phase orchestrator; `plan.ts` dedupes selected targets into a `MakePlan`; `registry.ts` holds the ordered target array; `signature.ts`/`applied.ts`/`scan.ts` implement semantic sync; `deploy.ts`/`deploy-store.ts` own the deploy store; `preflight.ts` validates embedded manifests.
- `src/provisioning/activation/` is the internal DSL: `contract.ts` (12-kind `Activation` union), `dispatch.ts` (exhaustive switches), `reconcile.ts` (multi-item envelope with structural per-item failure isolation, used by `defaults`, `duti`, `pipx`, `editorExtensions`, `release`), plus one runner file per kind. `file`, `tree`, `coderAgents`, `coderSkills`, `zedSettings`, `command`, `remoteInstaller` hand-roll their own envelopes.
- Capability modules (`src/pipx/`, `src/duti/`, `src/editor/`, `src/github/`, `src/defaults/`, `src/brew/`, `src/git/`, `src/coder/`, `src/zed/`) own external-tool protocols; they accept a `Context` and never import activation types. This boundary is currently clean (verified by import grep).
- `src/host/` owns `Context`, `CommandRunner`, `HostPath`, `mevRoot` (the `.mev` path authority), parsed-unknown narrowing (`parse.ts`), and atomic staging primitives (`transaction.ts`, `atomic-file.ts`, `directory-replacement.ts`, `symlink.ts`).
- `src/app/` orchestrates use cases (identity, config selection) with clean dependency inversion: no clipanion or host leakage, prompt/selection ports implemented in `src/cli/tty/`.
- `src/assets/` wraps `registry.generated.ts`, produced by `scripts/generate-assets.ts` from `src/assets/config/` (gitignored, regenerated by `pre*` hooks and by builds).
- `scripts/build.ts` and `scripts/install-mev.ts` are the release-compile and dev-install pipelines; `install.sh` is the end-user binary installer.

### Data flow

Selectors → `planMake()` → preservation hooks → applied-marker invalidation → `deployRole()` per role → `installPackages()` batch → `runActivation()` per target group in declaration order → per-group applied-signature write on success → `MakeReport` → `cli/tty/makelog.ts` rendering. `sync` prepends `scanTargets()` (concurrent signature + role-drift comparison) and feeds only stale targets into the same `runMake()`.

### External dependencies

clipanion 4.0.0-rc.4 (CLI), @inquirer/checkbox (toggle prompt), listr2 (live task list), js-yaml, tty-prog (first-party, tag-pinned), mermaid 11.16.0 and playwright-core 1.61.1 (both exact-pinned; the mermaid deep import `mermaid/dist/mermaid.min.js` and the `--external chromium-bidi/*` build flag are documented coupled workarounds in docs/architecture.md). External binaries driven at runtime: brew, git, gh, curl, shasum, defaults, duti, pipx, pandoc, pdftotext, Google Chrome, editor CLIs.

### Verification commands

- `bun run fix` — Biome auto-fix; always run before `check`.
- `bun run check` — Biome validation + `tsc --noEmit` (pre-hook regenerates assets and validates them).
- `bun run test` / `bun run test:unit` (src, hermetic) / `bun run test:integration` (tests/, sandboxed fs and fakes).
- `bun run build` — compiles the standalone binary.
- `bun e <command>` — run the CLI from source.

### Why the current state is insufficient

The audit (five parallel reviews over provisioning core, activation DSL + capabilities, CLI + app layers, internal/assets/scripts/CI, and the test suite) confirmed the strengths — clean capability/activation dependency direction, consistent error-taxonomy wrapping across all 19 commands, disciplined ENOENT handling, strong command-argument validation — and the debt inventory below. All groups are for implementation. Group D items were originally design questions; the user resolved each on 2026-07-20 and the recorded decisions below are binding.

## Technical Debt Inventory

Every item lists: Location, Why (reasoning), Impact (scope), Risk (of the change), Fix, Verify.

### Group A — Correctness and safety defects (implement now)

#### A1. Remote installers execute unverified when `checksumUrl` is omitted

- Location: `src/provisioning/activation/remote-installer.ts:74` (`if (!activation.checksumUrl) return;`), `src/provisioning/activation/contract.ts` (`checksumUrl?` optional field).
- Why: integrity verification of downloaded installer scripts is silently skipped when the optional field is absent; the type makes unverified remote code execution the easy default, violating the no-silent-fallback rule.
- Impact: every current and future `remoteInstaller` target declaration.
- Risk: low; some upstream installers publish no checksum, so an explicit escape hatch is needed.
- Fix (decided 2026-07-20): replace the optional field with a required discriminant `integrity: { checksumUrl } | { acknowledgedUnverified: true }` so the unverified path is a loud, reviewed declaration. Update all five `remoteInstaller` declarations — `rust.ts` (currently has `checksumUrl`), `bun.ts`, and `coder.ts` (×3, currently none) — to the new field. Because `checksumUrl` already feeds the target signature (`signature.ts:55,171`), replacing it changes the signature projection: all five targets legitimately resync once (state this in the change description, per the Constraints).
- Verify: `tests/provisioning/remote-installer.test.ts` gains cases for both branches; `bun run test:integration`.

#### A2. Release-version CI gate can silently no-op

- Location: `scripts/validate-release-version.js:26` (`if (import.meta.main)`), `.github/workflows/release.yml` `validate-release` job (bare `node`, no pinned Node setup).
- Why: `import.meta.main` is only defined on recent Node; on an older runner Node it is `undefined`, the guard block never runs, and the script exits 0 — the tag/version check passes vacuously.
- Impact: release integrity; a mismatched tag could publish.
- Risk: low.
- Fix: replace the guard with a check that works on all supported Node versions (for example comparing `process.argv[1]` to the module URL), or run the script with Bun/pinned Node in that job.
- Verify: existing `tests/scripts/validate-release-version.test.ts` still passes; manually run the script under `node` with a mismatched `GITHUB_REF_NAME` and confirm non-zero exit.

#### A3. Target-completion line misaligns for names longer than 7 characters

- Location: `src/cli/tty/makelog.ts:177-179` (`padEnd(7)`, magic width `22`, plain-width measurement via a reconstructed `✓` string).
- Why: registered targets include `antigravity_ide` (15 chars), `formulae` and `rust-cli` (8); for these the summary column silently misaligns, and the width measurement duplicates what `table.ts` does correctly on raw cells.
- Impact: `make`/`create`/`sync` output rendering.
- Risk: low.
- Fix: derive the name-column width from the widest target name in the report (the raw-cell measurement approach `table.ts` already uses); remove the magic constants.
- Verify: `src/cli/tty/makelog.test.ts` case using a long target name asserts aligned columns.

#### A4. `zedSettings` mutates the host before failing on unknown manifest entries

- Location: `src/provisioning/activation/zed.ts:64-83`.
- Why: `buildSettings` (file write) and `placeSymlink` run before the unknown-entry check, so a `failed` activation has already applied partial state; validation-before-mutation is inverted relative to the reconcile envelope's declare-first model.
- Impact: `zed` target activation.
- Risk: low.
- Fix: resolve the selection and fail on unknown entries before building or linking.
- Verify: `tests/provisioning/zed.test.ts` asserts no settings write and no symlink when unknown entries are present.

#### A5. `CommandRead.validate` return value is silently discarded

- Location: `src/provisioning/activation/contract.ts:92-97` (returns `unknown`), `src/provisioning/activation/command.ts:106-111` (result ignored; validation runs on the raw text while the binding is `trim()`ed).
- Why: the signature promises a producing validator but only the throw side-effect is honored; a transforming validator misbehaves silently, and pre-trim validation vs post-trim binding skew is unstated.
- Impact: every `command` activation using typed reads.
- Risk: low.
- Fix: type it `(raw: string, path: string) => void` and validate the same trimmed value that gets bound, or honor the return value as the binding — pick one and document it in `contract.ts`.
- Verify: `tests/provisioning/command.test.ts`; `bun run check`.

#### A6. Hardcoded `.mev` literals outside the `mevRoot` authority

- Location: `src/provisioning/targets/coder.ts:102,105,109`, `src/provisioning/targets/shell.ts:30`.
- Why: AGENTS.md declares `mevRoot` (`host/path.ts`) the sole authority and claims no host path hardcodes `.mev`; these four call sites contradict the stated invariant, so a root rename would silently miss them.
- Impact: coder and shell targets; the documented invariant's trustworthiness.
- Risk: low. Note: the deployed symlink destinations do not enter target signatures via these strings' location, but confirm signatures are unchanged after the mechanical rewrite (identical resulting paths).
- Fix: export a path builder from `host/path.ts` (for example `mevPath(...segments)` composing on `mevRoot`) and route these call sites through it.
- Verify: `rg "'\.mev" src --glob '!src/host/path.ts' --glob '!*.test.ts'` returns nothing outside generated/prose (note: the glob must be `!src/host/path.ts` — `!host/path.ts` does not match the file's repo-relative path and leaves the `mevRoot` authority line in the results); `bun run test`. Shipped asset-config content under `src/assets/config/` (shell/JSON consumer configs) legitimately references `.mev` and is out of scope.

#### A7. `sync` scan is all-or-nothing on a single target error

- Location: `src/provisioning/scan.ts:143-158` with `src/host/task-pool.ts` (`Promise.all` over workers; one thrown scan rejects the whole batch).
- Why: one unreadable applied marker or role directory aborts the entire scan with a single target's message, discarding the other targets' results; scan is read-only, so partial results are safe and more diagnosable.
- Impact: `sync` robustness.
- Risk: low-medium (changes `scanTargets`' error contract; callers must render per-target scan errors).
- Fix: settle per target and include a per-target error variant in `TargetScan`, surfacing failures without hiding healthy targets.
- Verify: `tests/provisioning/scan.test.ts` case where one target's marker is unreadable and others still classify.

### Group B — Structural duplication and ownership (implement now)

#### B1. Activation error boundary and status aggregation copied across runners

- Location: identical `catch → { ...base, status: 'failed', error: errorMessage(error) }` tails in `symlink.ts:130,177`, `coder.ts:162,190`, `zed.ts:86`, `command.ts:186`, `remote-installer.ts:192`, `reconcile.ts:74`. Two distinct status expressions exist, and they are NOT the same shape: the entry-status precedence `failed ? 'failed' : changed ? 'changed' : 'unchanged'` appears only at `reconcile.ts:71` and `command.ts:183`; the sites at `zed.ts:84`, `coder.ts:156,184` are a different 2-way boolean `built || linked ? 'changed' : 'unchanged'` over local flags, not entry-status aggregation.
- Why: the per-activation error boundary is structural only for reconcile kinds; hand-rolled kinds maintain it by copy-convention and have already drifted (zed's failed-with-entries shape).
- Impact: all 7 non-reconcile kinds; any future cross-cutting concern (timing, tagging) needs 8 edits.
- Risk: low; mechanical wrap.
- Fix: extract a shared guard (for example `guarded(base, fn): Promise<ActivationReport>`) exported beside the reconcile envelope; every hand-rolled runner wraps its body in it. Separately extract `aggregateStatus(entries)` for the two true entry-status sites (`reconcile.ts:71`, `command.ts:183`) only; do NOT force the `built || linked` boolean expressions in zed/coder through it, as they aggregate local flags, not entry statuses.
- Verify: existing `tests/provisioning/*.test.ts` suites pass unchanged.

#### B2. `runMake()` fuses orchestration, reporting, persistence, and presentation

- Location: `src/provisioning/run.ts:129-249` (121-line function); `formatBlocker`/`blockerReason` at `run.ts:71-80` consumed by `cli/tty/makelog.ts`; deploy/migration failures keyed by role (`run.ts:145-171`) while blockers attribute per target.
- Why: five concerns change for independent reasons in one function; blocker string formatting is a rendering concern living in the phase engine; role-keyed failure bookkeeping misattributes when targets share a role and discards error causes via `errorMessage()`.
- Impact: every provisioning command routes through this function; it is the hardest file to modify safely.
- Risk: medium; behavior must be preserved — the existing `tests/provisioning/run.test.ts` (444 lines) is the safety net.
- Note: `formatBlocker` is not purely presentational today — via `blockerReason` (`run.ts:214`) its output feeds `blockedReport(activation, reason)`, so the reason string enters the report model, not just the rendering. Moving `formatBlocker` to the TTY layer must either keep the report-model reason produced in the engine (a plain non-rendering string) or preserve the exact blocked-reason text; assert the blocked-report shape is unchanged.
- Fix: extract a deploy-phase helper (including the `migrateLegacySymlinks` loop) and a `computeBlockers(group, failedRoles, failedPackages)` function; key failures by target name and preserve the underlying error; move `formatBlocker` to the TTY layer next to its consumer. `runMake` becomes a phase sequencer.
- Verify: `tests/provisioning/run.test.ts` and `src/cli/tty/makelog.test.ts` pass; no report-shape change.

#### B3. `makelog.ts` mixes status classification with rendering

- Location: `src/cli/tty/makelog.ts` — classification (`isBlocked`, `targetStatus`, `countChanged`, `changedSummary`, `failedSummary`, `blockedSummary`, `summarizeActivationGroup`) interleaved with ANSI renderers; `summarizeActivationGroup` is exported only for its test.
- Why: "what happened" (report-model reduction) and "how it looks" are one file; tests reach into internals because the seam is missing.
- Impact: `provisioning.ts` and `activation-progress.ts` consumers.
- Risk: low-medium; mechanical split.
- Fix: move classification/summarization into a provisioning-owned report model (natural companion to the B2 extraction); `makelog.ts` consumes a typed status + summary and only renders. Do together with A3.
- Verify: restructured `makelog.test.ts` asserts classification and rendering separately.

#### B5. Coder runners are near-clones with no per-item failure isolation

### Group D — Decided design changes (implement now; user decisions recorded 2026-07-20)

#### D1. Declarative command steps replace `intentVersion` (decision: declarative step data)

- Location: `src/provisioning/activation/contract.ts` (`CommandStep` thunks `argv`/`env`/`skipIf` at `contract.ts:117-119`, `intentVersion` field at `contract.ts:59`), `command.ts` (factory validation, step execution, `readBindings`), `signature.ts:152-165` (the command intent projection specifically; 140-151 are the coderSkills/zedSettings cases), and the seven command targets (`nodejs`, `python`, `ruby`, `pnpm`, `coder`, `bun`, `rust` — note `rust.ts` is the command target; `rust-cli.ts` is a `release` kind and is out of scope).
- Why: signatures hash labels, captures, reads, and `changedWhen` but not the thunk bodies; the manual `intentVersion` counter (universally `1`, never bumped) is the only lever that surfaces implementation changes to `sync`, and forgetting it silently drifts the machine from intent.
- Impact: the command DSL contract and every command target; all command targets legitimately resync once after migration.
- Risk: highest in this plan; the rewrite must preserve each step's resolved argv/env behavior exactly, verified against the existing command and target tests.
- Fix: make steps declarative enough to hash — argv as a sequence of literal-or-named-reference tokens, env as a record of literal-or-reference values (references resolve from `reads`/`capture` scope as today), `skipIf` already declarative. Include the declarative step data in the signature intent projection; delete `intentVersion` from the contract, the factory validation, the signature, and every target. Sequence together with B19, which rewrites the same target files.
- Verify: `src/provisioning/signature.test.ts` gains cases proving an argv or env edit flips the signature; `tests/provisioning/command.test.ts` proves resolved behavior is unchanged; `rg intentVersion src tests` returns nothing.

#### D2. Dispatch switches stay (recorded decision — no implementation)

#### D3. Pin the CI runner image and gate asset shell scripts (decision: pin + shellcheck)

- Location: `runs-on: macos-latest` in `run-static-checks.yml`, `run-tests.yml`, `run-build.yml`, `release.yml`; five `.sh` files under `src/assets/config/` ship unchecked.

- Impact: whole CI determinism; asset-script quality.
- Risk: low.
- Fix: pin an explicit macOS image version (current stable, for example `macos-15`) in every workflow with a comment stating bumps are deliberate; add a shellcheck step to `run-static-checks.yml` scoped to `src/assets/config/**/*.sh` (preinstalled on GitHub macOS runners; `brew install shellcheck` as fallback). - Verify: CI green on the pinned image; a deliberately planted shellcheck violation fails the static-checks workflow.

## Constraints

- Invariants that must hold after every change: capability modules import no activation types; activations may import capabilities, never the reverse; multi-item kinds keep structural per-item failure isolation; all parsed-unknown data narrows through `host/parse.ts`; every mev-managed host path derives from `mevRoot`; `CommandLineError` → stdout with usage, `AppError`/`ProvisioningError` → stderr exit 1 via `runReportingDomainErrors`; no silent fallbacks.
- Target signatures are user-visible state: any change to activation declarations (labels, reads, captures, `changedWhen`, packages, assets) reselects targets on the next `sync`. Refactors must either preserve declarations byte-identically or state the expected one-time resync.
- Test layering: pure logic colocated under `src/`, anything touching fs/CLI-routing/subprocess/network under `tests/` with the shared fixtures; expected values come from independent oracles, never the implementation under test.
- Naming rules: no `base`/`common`/`core`/`utils`/`helpers` modules; file and symbol names must not repeat their directory context; new shared code gets a specific responsibility name.
- Refactors delete the superseded copies in the same change; comprehensive `rg` sweeps confirm no remnants of renamed or removed structures.
- Documentation follows implementation: update AGENTS.md, docs/architecture.md, and docs/testing.md where behavior or structure they describe changes (the `.mev`-authority claim becomes true again after A6; the reconcile-kind table changes with B5). Declarative style, English, no bold emphasis.
- The Group D decisions recorded on 2026-07-20 are binding: D1/D3/D4/D5/D6 are in scope for implementation as specified; D2 (descriptor table) is rejected and must not be implemented. Deviating from a recorded decision requires new user sign-off.
- D1 supersedes B19's label-preservation caution for the command targets: their one-time resync is expected and stated; all other refactors still preserve activation declarations byte-identically.
- `bun run fix` runs before `bun run check`; test commands run only for changes with runtime surface.
